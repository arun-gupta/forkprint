# Feature Specification: `claude-worktree.sh --cleanup-merged` / `--remove` run from inside a worktree

**Feature Branch**: `278-claude-worktree-sh-allow-cleanup-merged`
**Created**: 2026-04-16
**Status**: Draft
**Related issue**: #278
**Input**: Teach `scripts/claude-worktree.sh --cleanup-merged` and `--remove` to work when invoked from inside a worktree. (1) If the user omits the issue-number argument and the current branch begins with `^[0-9]+-`, infer the issue number from the branch name. (2) If the current working directory is inside a non-primary git worktree (detectable via `git rev-parse --git-common-dir` ≠ `git rev-parse --git-dir`), resolve the main repo path, `cd` there, check out `main`, then run the existing cleanup flow. (3) When the worktree being removed *is* the caller's CWD, emit a clear final-line notice telling the user to `cd <main-repo-path>` because their shell's directory no longer exists. Keep today's behavior from the main repo unchanged. Keep the existing `gh pr view --json state == MERGED` merge-status guard as the sole gate on `--cleanup-merged` — no interactive "are you sure?" prompt.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Self-cleanup from a headless worktree Claude session after a PR merges (Priority: P1)

As a RepoPulse maintainer who spawned a Claude session via `scripts/claude-worktree.sh <issue>` to work on a specific GitHub issue, once the PR for that issue is merged, I want to run `scripts/claude-worktree.sh --cleanup-merged` from inside the same worktree — with no argument — and have the script correctly infer the issue number from my current branch, `cd` itself to the main repo, pull `main`, remove the worktree, and delete the feature branch. Today the Claude session that just finished the PR cannot trigger its own cleanup: the script requires the explicit issue number, requires being run from the main repo, and requires that repo to already be on `main`. The session has to stop and tell the user to open a new terminal and run the cleanup manually. Eliminating that friction is the whole point of this feature.

**Why this priority**: This is the only user story in issue #278. Every merged worktree-scoped PR hits this friction today, and the volume will only grow as more issues are worked in parallel worktrees. P1 because it is the feature.

**Independent Test**: In a worktree created by `scripts/claude-worktree.sh <issue>`, after the PR for `<issue>` is merged on GitHub, running `scripts/claude-worktree.sh --cleanup-merged` (no argument) from inside that worktree's directory completes successfully — the worktree is removed, the feature branch is deleted, the main repo is left on `main` with latest upstream, and the caller's shell is shown a clear one-line notice stating that the CWD no longer exists and which path to `cd` into.

**Acceptance Scenarios**:

1. **Given** a worktree at `../forkprint-278-claude-worktree-sh-allow-cleanup-merged` on branch `278-claude-worktree-sh-allow-cleanup-merged` whose PR is `MERGED` on GitHub, **When** a shell inside that worktree runs `scripts/claude-worktree.sh --cleanup-merged` with no issue argument, **Then** the script: (a) infers issue number `278` from the branch name, (b) resolves the main repo path by walking `git worktree list`, (c) `cd`s to the main repo, (d) checks out `main` if it is not already there, (e) `git pull --ff-only origin main`, (f) kills any `.dev.pid` / `.claude.pid` processes recorded under the worktree, (g) `git worktree remove --force` the worktree, (h) `git branch -D 278-claude-worktree-sh-allow-cleanup-merged`, (i) exits 0 with a clearly labeled final line pointing the user to `cd <main-repo-path>` because their previous CWD no longer exists.
2. **Given** the same worktree is on branch `my-hand-rolled-branch` (no numeric prefix) and the user runs `scripts/claude-worktree.sh --cleanup-merged` with no argument, **When** the script inspects the current branch, **Then** it exits non-zero with an error that names both the problem (no numeric prefix in branch name, cannot infer issue) and the fix (re-run with an explicit issue number, `scripts/claude-worktree.sh --cleanup-merged <issue>`), and performs no destructive action.
3. **Given** a shell sitting in the main repo (not a worktree) on branch `main`, **When** the user runs `scripts/claude-worktree.sh --cleanup-merged` with no argument, **Then** the script exits non-zero with the existing usage error (`Usage: ... --cleanup-merged <issue>`) — no-arg inference MUST NOT kick in outside a worktree. This preserves today's behavior for every user who runs the script from the main repo.
4. **Given** a shell inside a worktree on branch `264-investigate-skipped-test-per-repo-status` whose PR is `OPEN` (not merged), **When** the user runs `scripts/claude-worktree.sh --cleanup-merged` with no argument, **Then** the script infers issue `264`, probes PR state via `gh pr view`, sees `OPEN`, refuses the cleanup, points to `--remove`, and exits non-zero without removing anything. The existing merge-status guard must fire the same way whether or not the issue number was inferred.
5. **Given** a shell inside a worktree whose branch is `278-…` and whose PR is `MERGED`, **When** the user runs `scripts/claude-worktree.sh --cleanup-merged 278` (explicit arg), **Then** the script performs the same successful cleanup as scenario 1 — explicit arg from inside a worktree MUST keep working and MUST NOT be broken by the new inference path.

### User Story 2 — `--remove` works the same way from inside a worktree (Priority: P2)

As a maintainer who wants to discard an unmerged worktree from the inside (the Claude session decided the work is abandoned, or a spec review concluded it's not worth pursuing), I want `scripts/claude-worktree.sh --remove` with no argument to behave the same way as `--cleanup-merged` with no argument: infer the issue from the branch, resolve the main repo, perform the force-remove, and tell the caller their CWD is gone. `--remove` deliberately skips the merge-status guard, but the worktree-to-main-repo relocation logic is identical and MUST be shared with `--cleanup-merged`.

**Why this priority**: P2 because `--remove` is the escape-hatch path, used less often than `--cleanup-merged`, but ergonomically it must match or users will have to mentally track two different invocation rules for two near-identical commands. Consistency pays itself back every time.

**Independent Test**: In a worktree on a branch like `999-abandoned-experiment`, running `scripts/claude-worktree.sh --remove` (no arg) from inside the worktree force-removes it and leaves the caller's shell in a stranded state with a clear final-line notice — regardless of PR state.

**Acceptance Scenarios**:

1. **Given** a shell inside a worktree on branch `999-abandoned-experiment` with no associated PR or a `CLOSED` / `OPEN` PR, **When** the user runs `scripts/claude-worktree.sh --remove` (no argument), **Then** the script infers issue `999`, resolves the main repo, kills any recorded dev/claude PIDs, `git worktree remove --force`s the worktree, prints a final-line stranded-shell notice, and exits 0. PR state is NOT consulted (matches today's `--remove` semantics).
2. **Given** a shell inside a worktree on branch `random-name-no-number`, **When** the user runs `scripts/claude-worktree.sh --remove` (no argument), **Then** the script exits non-zero with an error that names the problem and the fix — identical message shape to User Story 1 scenario 2.
3. **Given** a shell in the main repo (not a worktree), **When** the user runs `scripts/claude-worktree.sh --remove` (no argument), **Then** the existing usage error fires — no-arg inference does NOT kick in outside a worktree.

---

### Edge Cases

- **Main-repo primary worktree is on a non-`main` branch when `--cleanup-merged` is invoked from inside a worktree**: today the script refuses and tells the user to switch the primary worktree to `main` first. The new inside-a-worktree path MUST either (a) check out `main` in the primary worktree as part of the inferred flow, or (b) preserve today's refusal with the same error. The feature prefers (a) — auto-checkout `main` in the primary worktree — because the whole point is to eliminate manual steps. If there are uncommitted or unstaged changes in the primary worktree that would prevent a `git checkout main`, the script MUST NOT force-discard them; it MUST refuse with a clear error naming the dirty path. Clean primary-worktree state is an assumption, not a precondition to destroy.
- **Main repo cannot be resolved** (corrupt git state, `git worktree list` empty, the primary worktree path no longer exists on disk): the script MUST exit non-zero with a message that names the failure and does NOT delete anything. Silent fallback to today's "run from the main repo" path is not acceptable when the inference layer has already bound the destructive intent to a specific worktree.
- **The worktree being cleaned up is the caller's CWD** (the common case this feature exists for): the script MUST `cd` out of that directory before issuing `git worktree remove`; if it doesn't, macOS/Linux will still let `git worktree remove` succeed, but subsequent shell commands (including the final-line print) may behave unpredictably in the stranded directory. The script MUST `cd` to the resolved main-repo path first, perform the remove, then print the stranded-shell notice naming that same path so the user's next command can be `cd <printed-path>`.
- **The caller's CWD is a worktree, but the inferred issue number points to a *different* worktree** (e.g., user is in `forkprint-264-...` on a branch someone renamed to start with `278-`): the branch-name inference trusts the branch, not the directory path. The script removes the worktree associated with the *inferred issue number*. If that is not the caller's CWD, the stranded-shell warning is unnecessary and MUST be suppressed — only emit the warning when the removed worktree path equals the caller's CWD.
- **Shell is inside the `.git/worktrees/...` admin path, not the worktree working tree**: `git rev-parse --git-common-dir` vs `git rev-parse --git-dir` detection handles this correctly — both values diverge inside any worktree context (primary or linked). The primary worktree is the one where `git-common-dir` equals `git-dir`'s parent. The script MUST treat "inside a linked worktree" (the common case) and "inside a primary worktree" (the main repo) as distinct and behave differently: inference only fires in the former.
- **Branch name matches `^[0-9]+-` but the number is not a real GitHub issue** (e.g., `99999-test`): inference still fires and the cleanup flow will reach the existing `gh pr view` call, which will fail with "no PR found for branch". The script MUST surface that existing error — this feature does not add a pre-flight "is this a real issue number" check, since the current `gh pr view` guard already handles the failure mode correctly.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `scripts/claude-worktree.sh --cleanup-merged` MUST accept a no-argument invocation when the caller's current working directory is inside a linked (non-primary) git worktree AND the current branch name matches `^[0-9]+-`. In that case the script MUST infer the issue number from the leading digits of the branch name and proceed as if the user had typed that number as an explicit argument.
- **FR-002**: `scripts/claude-worktree.sh --remove` MUST accept a no-argument invocation under the identical condition described in FR-001 and MUST use the same inference rule.
- **FR-003**: Detection of "inside a linked worktree" MUST use `git rev-parse --git-common-dir` vs `git rev-parse --git-dir` (they are equal inside the primary worktree and divergent inside a linked worktree). It MUST NOT rely on pattern-matching the current directory path against `forkprint-*` or any other filesystem convention.
- **FR-004**: When invoked with no argument from a shell whose current branch does NOT match `^[0-9]+-`, the script MUST exit non-zero with an error that (a) names the problem ("cannot infer issue number from branch `<branch>`") and (b) tells the user the fix ("re-run with `scripts/claude-worktree.sh --cleanup-merged <issue>` or `--remove <issue>`"). No destructive action may be taken.
- **FR-005**: When invoked with no argument from a shell that is NOT inside a linked worktree (i.e., the primary worktree / main repo clone, OR a non-git directory), the script MUST preserve today's usage-error behavior and MUST NOT attempt branch-name inference. This protects users who run the script from the main repo.
- **FR-006**: Before performing any destructive operation from inside a worktree, the script MUST resolve the main repo path (the primary worktree) by reading `git worktree list --porcelain` and selecting the entry whose `worktree` line points to the directory containing `.git` as a real directory (not a gitdir file). It MUST `cd` into that main-repo path before invoking `git worktree remove`, `git branch -D`, or `git pull`.
- **FR-007**: When the worktree being removed is the caller's current working directory, the script's final output line MUST be a clearly labeled notice of the form `note: your shell's previous CWD (<removed-path>) no longer exists — run \`cd <main-repo-path>\` to continue`. This is the stranded-shell warning. It MUST be the last thing printed before exit so a user scanning the tail of the output sees it.
- **FR-008**: When the removed worktree is NOT the caller's current working directory (inference fired but the caller happened to be elsewhere), the stranded-shell warning MUST be suppressed — it would be inaccurate and noisy.
- **FR-009**: The existing merge-status guard in `--cleanup-merged` (`gh pr view --json state` must report `MERGED` for the inferred branch) MUST run unchanged. No new interactive "are you sure?" prompt may be introduced on any path. `--cleanup-merged`'s safety comes from the PR state check; `--remove`'s lack of safety is the existing design and stays as-is.
- **FR-010**: If the primary worktree is not on `main` when `--cleanup-merged` is invoked from inside a linked worktree, the script MUST check out `main` in the primary worktree automatically — this is the ergonomic core of the feature. If `git -C <main-repo> checkout main` fails (uncommitted changes, merge in progress, etc.), the script MUST exit non-zero with an error naming the dirty path and MUST NOT force-discard state with `git checkout -f`, `git reset --hard`, `git stash`, or any other state-destroying command.
- **FR-011**: `scripts/claude-worktree.sh --help` output MUST document the new no-argument invocations for both `--cleanup-merged` and `--remove`, including the branch-name-inference rule and the stranded-shell warning. The existing explicit-argument usages MUST remain documented — this feature adds a form, it does not replace one.
- **FR-012**: `docs/DEVELOPMENT.md` "Cleanup" subsection MUST be updated to show both flows side by side — the from-main-repo form (today) and the from-inside-a-worktree form (new). The update MUST include a one-line note on the stranded-shell warning so maintainers reading the docs know to expect it.
- **FR-013**: The feature MUST NOT change the behavior of `--approve-spec` or `--revise-spec`. Those already work cross-worktree via recorded session UUID and `claude -p --resume` and are outside this feature's scope.
- **FR-014**: The feature MUST NOT introduce any new runtime dependency beyond what `scripts/claude-worktree.sh` already requires (`git`, `gh`, standard POSIX utilities already in use, bash features already used by the script). No new binary, no new language, no new configuration file.
- **FR-015**: `--cleanup-merged` invoked with an explicit issue-number argument from inside a worktree MUST continue to work exactly as it does today — the argument wins over inference when both are available. Inference only fills the gap left by a missing argument.
- **FR-016**: `--remove` invoked with an explicit issue-number argument from inside a worktree MUST likewise continue to work exactly as it does today.
- **FR-017**: The full cleanup sequence MUST remain: (1) verify merge (for `--cleanup-merged` only), (2) `git pull --ff-only origin main` in the primary worktree, (3) kill any recorded `.dev.pid` / `.claude.pid`, (4) `git worktree remove --force <wt>`, (5) for `--cleanup-merged` only, `git branch -D <branch>`. The from-inside-worktree form changes *where* the script runs, not *what* the script does.

### Key Entities

This feature has no data model. The shell-level entities it reasons about already exist:

- **`scripts/claude-worktree.sh`** — the bash script being extended. The two functions `cleanup_merged()` (line 86) and `remove_worktree()` (line 67) are the extension points. A new shared helper — call it `resolve_worktree_context()` — likely belongs alongside them to concentrate the "am I inside a linked worktree?", "what is the main repo path?", and "what issue number does the branch imply?" logic in one testable block.
- **Linked worktree** — any directory registered under the primary repo's `.git/worktrees/` admin area. The Claude-per-issue worktrees created by `scripts/claude-worktree.sh <issue>` are all linked worktrees; the main clone is the primary worktree. The feature only triggers inference in linked worktrees.
- **Primary worktree (main repo)** — the clone that owns the `.git` directory. Destructive cleanup operations run from here.
- **Branch name** — the string returned by `git rev-parse --abbrev-ref HEAD` inside the linked worktree. The inference rule extracts the leading run of digits (`^([0-9]+)-.*`).
- **`docs/DEVELOPMENT.md` "Cleanup" subsection** — the docs block that must grow to document both flows.
- **`scripts/claude-worktree.sh --help`** — the usage block inside `print_usage()` in the script; must gain entries for the no-argument forms.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A maintainer can complete post-merge cleanup from inside a worktree with exactly one command (`scripts/claude-worktree.sh --cleanup-merged`, no argument) where today they need a minimum of four discrete shell commands executed in a second terminal: `cd <main-repo>`, `git checkout main`, `git pull`, `scripts/claude-worktree.sh --cleanup-merged <issue>`. Reduction: 4 → 1.
- **SC-002**: Post-merge cleanup no longer requires switching terminal windows. The same shell that ran the Claude session can run cleanup. Measured: zero new terminals opened for cleanup on 100% of merged worktree-scoped PRs on this maintainer's machine across the next five merges.
- **SC-003**: Running the no-argument form outside a worktree (in the main repo clone) produces the same usage error it does today with the same exit code and message shape, verified by a direct string comparison between the pre-change and post-change error output.
- **SC-004**: The existing merge-status guard still fires correctly on `OPEN`, `CLOSED`-without-merge, and missing-PR states — no regression in safety. Verified by running `--cleanup-merged` (no arg, from inside a worktree) against a worktree whose PR is `OPEN` and confirming the script refuses and exits non-zero without deleting anything.
- **SC-005**: The stranded-shell warning appears on 100% of successful cleanups where the caller's CWD was the removed worktree, and does NOT appear when the caller's CWD was elsewhere. Verified by two end-to-end shell sessions.
- **SC-006**: `scripts/claude-worktree.sh --help` output and the `docs/DEVELOPMENT.md` "Cleanup" subsection both describe both flows (from-main-repo and from-inside-a-worktree). Verified by grep on the keywords `--cleanup-merged` in both files showing the new forms alongside the old.

## Assumptions

- The primary worktree (main repo) is in a clean, committable state when the maintainer triggers `--cleanup-merged` from a linked worktree. Dirty primary-worktree state is rare in this workflow (the maintainer works in worktrees precisely to keep `main` clean), but the script MUST refuse rather than destroy if it is encountered. Cleanup must never fight the maintainer's in-flight work on the main clone.
- `git rev-parse --git-common-dir` and `git rev-parse --git-dir` are available and reliable on the git versions already required by the repo. These are core git plumbing commands available since well before any git version this project supports.
- The `^[0-9]+-` branch-name convention is the only inference rule this feature implements. Non-digit-prefixed branches (personal branches, hotfix branches, experiment branches) explicitly opt out — the maintainer passes `<issue>` explicitly in those cases. This is acceptable because the `scripts/claude-worktree.sh <issue>` spawn path always produces digit-prefixed branches by construction; a linked worktree on a non-digit-prefixed branch was created by hand and the maintainer already knows the issue number.
- `gh pr view <branch> --json state` will find the merged PR for a branch name that matches the digit-prefix convention, because those branches are the same ones `scripts/claude-worktree.sh` spawned and pushed. No new `gh` behavior is assumed.
- The existing behavior of killing `.dev.pid` and `.claude.pid` during cleanup is unchanged by this feature. These files are recorded inside the linked worktree directory and can be read before the worktree is removed.
- The script is already guarded by `set -euo pipefail` and uses `git -C <path>` to target specific worktrees — the new code will follow those same conventions. No new shell-safety patterns need to be invented for this feature.
