# RepoPulse Scoring Methodology

This document explains how RepoPulse scores repositories, where the scoring thresholds come from, and how to reproduce or refresh the calibration data.

---

## Why calibrated thresholds?

Earlier versions of RepoPulse used hardcoded thresholds with no empirical basis — a repo with 10k stars was labeled "Growing" (a middle tier) when it is actually in the top 1–2% of all GitHub repositories. The distribution of stars, fork rates, and activity metrics across GitHub is extremely skewed, and arbitrary thresholds produce misleading scores.

Calibrated thresholds anchor scores to real observed distributions, so a "Strong" rating means the repository genuinely outperforms most comparable repositories — not just that it crossed an editorial guess.

---

## Scoring anchor: star brackets

All scores are computed relative to repositories in the same **star bracket**, not against the full GitHub population. Stars are used as the anchor because they are a reasonable proxy for project maturity and visibility — user expectations, contributor volume, and maintainer load differ meaningfully across star ranges.

| Bracket | Star range | Description |
|---|---|---|
| Emerging | 10 – 99 | Early-stage or niche projects with some external interest |
| Growing | 100 – 999 | Projects with meaningful adoption |
| Established | 1,000 – 9,999 | Well-known projects with active communities |
| Popular | 10,000+ | Widely adopted, high-visibility projects |

**Known limitation:** Stars correlate with maturity but are also influenced by marketing and virality. A single anchor metric will never be perfect. This is a pragmatic simplification chosen for explainability. Future calibration may stratify by additional dimensions such as repo age or domain.

## Data source

Calibration data is generated by running the RepoPulse analyzer itself against a stratified sample of GitHub repositories.

**Why not BigQuery?**
We investigated `bigquery-public-data.github_repos` and GH Archive as primary sources. The `github_repos.repos` table (which contained star and fork counts) has been retired. GH Archive records events but not repo state — star counts, fork counts, and watcher counts do not appear in any event payload.

**Chosen approach:** GitHub Search API + RepoPulse GraphQL analyzer
- The GitHub Search API (`/search/repositories?q=stars:X..Y`) is used to sample repositories within each star bracket
- Each sampled repository is analyzed using the same RepoPulse GraphQL analyzer used for live analysis
- This guarantees that calibration data and live scores are derived from identical API fields and parsing logic — no cross-source mapping errors

---

## Sampling approach

For each star bracket, repositories are sampled using multiple sort strategies (`updated`, `created`, `stars`) across search result pages. Using multiple sort orders diversifies the sample beyond what a single ordered list would produce.

The GitHub Search API caps results at 1,000 per query. For brackets with very large populations (e.g., Emerging with millions of repos), the sample is a statistical sample, not a census. The minimum target is **50 repos per bracket** — the point where p25/p50/p75 estimates are stable and p90 has at least 5 anchor values. More is better where API rate limits allow. Sample sizes are recorded in the calibration output and disclosed in the UI.

See [Statistical approach](#statistical-approach-why-percentiles-not-mean--sd) for the reasoning behind the sample size target.

**Search filters applied to every bracket:**

| Filter | Value | Rationale |
|---|---|---|
| `fork:false` | — | Forks inherit parent history and skew contributor and commit distributions |
| `archived:false` | — | Archived repos have no ongoing activity; all responsiveness metrics would be stale |
| `pushed:>DATE` | See table below | Ensures the repo has had recent activity; filters abandoned projects |
| `stars:MIN..MAX` | Bracket-specific | See bracket table above |

**Pushed-after cutoff by bracket:**

| Bracket | Pushed after | Rationale |
|---|---|---|
| Emerging | 2025-01-01 | Tighter window — a 10–99 star repo inactive for 15+ months is almost certainly abandoned |
| Growing | 2024-10-01 | 18-month window gives enough history for responsiveness metrics |
| Established | 2024-10-01 | Same as Growing |
| Popular | 2024-10-01 | Same as Growing; large repos occasionally go quiet without being dead |

**Client-side quality filters (applied after fetching):**

The GitHub Search API index can be stale and does not support all necessary exclusion criteria. The following additional filters are applied in the listing script:

| Filter | Rationale |
|---|---|
| Primary language must not be null | Repos with no detected language are usually markdown-only list or resource repos |
| Excluded primary languages: `Jupyter Notebook`, `Adblock Filter List`, `TeX` | These indicate notebook collections, filter lists, or document repos — not software |
| Star count validated against bracket bounds | Search index staleness can return repos outside the intended range |
| Repo name ≠ owner name | Filters GitHub profile README repos (e.g. `user/user`) |
| Name does not end in `-docs`, `-documentation`, `-website`, `-wiki`, `-guidelines` | Filters documentation sites that are not the software project itself |
| Description does not contain mirror indicators | Filters read-only mirrors where PRs and issues are not the canonical activity stream |
| Description does not contain bypass/OTP/account-cycling indicators | Filters credential tools and security bypass repos with atypical activity patterns |
| Name and description do not match collection patterns | Filters "awesome" lists, curated link collections, cheatsheets, and roadmap repos |
| Max 3 repos per primary language per bracket | Enforces language diversity so no single language dominates the calibration sample |

**Sort strategy for candidate fetching:**

Candidates are fetched using three sort strategies (`updated`, `stars`, `created`) to diversify the sample. For the popular bracket, `updated` is prioritised over `stars` to reduce the influence of repos with artificially inflated star counts.

**Established bracket sub-range sampling:**

The established bracket (1k–10k stars) uses three sub-ranges (`1000–2999`, `3000–5999`, `6000–9999`) with ~7 candidates drawn from each. This prevents the sample from clustering at the top of the bracket.

**Minimum activity filter:** Repositories that pass all filters but have unavailable metrics (e.g., no issues or PRs in the analysis window) are not excluded from the sample — their available metrics still contribute to applicable percentiles.

## Metrics and how they are derived

For each sampled repository, the following metrics are collected and their percentile distributions computed within each star bracket:

| Metric | Source | Derivation |
|---|---|---|
| Stars | GraphQL `stargazerCount` | Direct |
| Forks | GraphQL `forkCount` | Direct |
| Watchers | GraphQL `watchers.totalCount` | Direct |
| Fork rate | Stars, Forks | `forks / stars` |
| Watcher rate | Stars, Watchers | `watchers / stars` |
| PR merge rate | Activity (90d window) | `prsMerged / prsOpened` |
| Issue closure rate | Activity (90d window) | `issuesClosed / issuesOpened` |
| Stale issue ratio | Activity (90d window) | Direct |
| Median time to merge | Activity (90d window) | Direct (hours) |
| Median time to close | Activity (90d window) | Direct (hours) |
| Issue first response (median) | Responsiveness metrics | Direct (hours) |
| Issue first response (p90) | Responsiveness metrics | Direct (hours) |
| PR first review (median) | Responsiveness metrics | Direct (hours) |
| Top contributor share | Commit author distribution | `top author commits / total commits` |

Percentiles computed: **p25, p50, p75, p90** within each star bracket.

## Statistical approach: why percentiles, not mean ± SD

### What we investigated

During calibration design we explored whether GitHub metrics follow a normal distribution — which would allow using mean ± standard deviation as scoring thresholds. We ran empirical rule checks (68%/95%/99.7%) on fork count data across stratified random samples in the Growing bracket (100–999 stars).

**Finding:** Fork counts — and GitHub metrics generally — follow a **log-normal distribution**, not a normal distribution. The raw data is heavily right-skewed with extreme outliers (e.g. a large org repo in the same star range as a personal project). Even with stratified sampling and balanced strata, one outlier can move the mean and SD dramatically.

### Transformations considered

| Transform | Skewness (example run) | Excess kurtosis | Notes |
|-----------|------------------------|-----------------|-------|
| None (raw) | 3.72 | 12.75 | Strongly right-skewed |
| sqrt(x) | 2.68 | 7.47 | Improved but still skewed |
| cbrt(x) | 2.03 | 4.84 | Better still |
| log(x) | 0.36 | 1.02 | Best — Box-Cox optimal λ=0 confirms this |

Log transformation brings skewness close to zero and dramatically reduces kurtosis. However, even log-transformed data does not reliably satisfy the empirical rule at 1σ because structural outliers (large org repos with atypically high fork counts for their star range) are an inherent feature of the GitHub population, not a sampling artifact.

### Why percentiles are the right choice

Percentiles are **distribution-agnostic** — they require no assumption of normality and are unaffected by the choice of transformation. They also degrade gracefully in the presence of outliers: an extreme outlier only affects the p90 estimate, leaving p25, p50, and p75 stable.

With mean ± SD thresholds, a single repo like `GoogleCloudPlatform/magic-modules` (2,236 forks in the Growing bracket) inflates the SD enough to make the 1σ band meaningless. With percentiles, it simply sits above p90 and does not distort the lower bands.

### Why not log-transform then compute percentiles

Log-transforming before computing percentiles adds complexity with no benefit — percentiles on raw data and percentiles on log-transformed data produce equivalent threshold rankings once converted back. Raw percentiles are simpler, more interpretable, and directly usable in the scoring functions.

### Sample size per bracket

The minimum sample size is driven by p90 stability, since p90 is anchored by the top 10% of values:

| Sample size | Data points anchoring p90 | Assessment |
|-------------|--------------------------|------------|
| 20 | 2 | Too unstable |
| 30 | 3 | Borderline |
| 50 | 5 | Acceptable for quarterly-refreshed calibration |
| 100 | 10 | Good |

**Target: 50 repos minimum per bracket**, with higher targets preferred where API rate limits allow. For the Emerging bracket, where many repos lack issue/PR history (producing `unavailable` metrics), a 2× buffer is applied — target 50 repos but expect usable responsiveness data from roughly half.

### Strata granularity

We also investigated whether finer strata (more subdivisions of the star range) would produce more normally distributed samples within each stratum. Finding: strata granularity has diminishing returns because stars and fork counts are only loosely correlated — a 900-star org repo can have 10× the forks of a 900-star personal project. Finer strata reduce within-stratum star variance but do not meaningfully reduce fork count variance. The current 4-stratum design (equal-width star bands per bracket) provides adequate diversity without over-engineering the sampling process.

## Calibration output

The calibration script writes a versioned JSON file to `lib/scoring/calibration-data.json`:

```json
{
  "generated": "2026-04-08",
  "source": "GitHub Search API + RepoPulse GraphQL analyzer",
  "sampleSizes": {
    "emerging": 487,
    "growing": 512,
    "established": 498,
    "popular": 203
  },
  "brackets": {
    "emerging": {
      "sampleSize": 487,
      "stars":    { "p25": 3,    "p50": 12,   "p75": 45,   "p90": 78   },
      "forkRate": { "p25": 0.02, "p50": 0.08, "p75": 0.18, "p90": 0.31 },
      "prMergeRate": { "p25": 0.21, "p50": 0.48, "p75": 0.71, "p90": 0.89 }
    }
  }
}
```

The `generated` date is the calibration cutoff date — analogous to an LLM training cutoff. Scores reflect the GitHub population as it existed on that date.

## Staleness policy

Calibration data is refreshed **quarterly**. If the `generated` date in `calibration-data.json` is more than **6 months old**, the RepoPulse UI surfaces a visible staleness warning:

> *"Scores calibrated against GitHub data from [date]. A more recent calibration is recommended."*

This ensures users are aware when scores may have drifted from current GitHub norms.

## How to re-run calibration

### Recommended: calibrate-fast (minutes, not hours)

`calibrate-fast` uses lightweight custom GraphQL queries and alias batching (5 repos per query) instead of the full `analyze()` pipeline. It skips paginated commit history — the main bottleneck in the original script — and supports multiple tokens for parallel rate limit capacity.

**Requirements:**
- Node.js 18+
- At least one GitHub PAT with `public_repo` read access in `.env.local`

**Single token:**
```
GITHUB_TOKEN=ghp_yourtoken
```

**Multi-token round-robin (recommended):**
```
GITHUB_TOKENS=ghp_token1,ghp_token2,ghp_token3
```

Add `GITHUB_TOKENS` (comma-separated PATs) to `.env.local`. The script round-robins across all tokens for every GraphQL and REST call. Each additional token adds roughly one full token's worth of rate limit capacity — 3 tokens yield approximately 3× throughput. `GITHUB_TOKENS` takes precedence over `GITHUB_TOKEN` when both are set.

```bash
npm run calibrate-fast
```

The script checkpoints to `scripts/calibrate-fast-checkpoint.json` after every batch. If interrupted, re-running resumes from where it left off. Delete the checkpoint file to start fresh.

**Expected runtime:** ~30–60 minutes for 50 repos × 4 brackets with a single token. With 3 tokens, typically under 20 minutes.

**Metrics not collected by calibrate-fast** (require bot detection / maintainer identification not available via public API):
`humanResponseRatio`, `botResponseRatio`, `contributorResponseRate`, `stalePrRatio`

### Original script (slow — kept for reference)

```bash
npm run calibrate
```

Checkpoints to `scripts/calibrate-checkpoint.json`. Uses the full `analyze()` pipeline with paginated commit history — approximately 10 repos/hour per token. At 50 repos × 4 brackets, expect ~20 hours. Not recommended for routine calibration refreshes.

After either run completes, commit the updated `lib/scoring/calibration-data.json` with a message referencing the run date and sample sizes.

---

## Quarterly refresh process

1. Delete `scripts/calibrate-fast-checkpoint.json` if one exists from a previous run
2. Add tokens to `.env.local` — the more the faster (see above)
3. Run `npm run calibrate-fast`
4. Review the output `lib/scoring/calibration-data.json` — verify sample sizes are ≥ 50 per bracket and percentile values are plausible
5. Commit the updated file with a message such as: `chore: recalibrate scoring thresholds (2026-Q2, N=200 repos)`
6. The UI staleness warning will clear automatically once the new `generated` date is within 6 months
